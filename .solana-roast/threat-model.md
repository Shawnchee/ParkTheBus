- **Launch Readiness** (governance/operational posture — branch 6 + branch 9 + process): `0C / 2H / 0M / 0L`

Read this as: **clean code, launch posture to finish.** A single collapsed score would have let the two

## Detailed Findings

### F-01 — No council/authority update or recovery path · HIGH · Launch Readiness
- **Where:** `initialize.rs:43` (authority recorded), `lib.rs:20-85` (no update instruction exists),
  `state.rs:71-80` (`Config` has no setter).
- **What:** `config.authority` is written at init and then never read by any handler. The council list
  and threshold are fixed at `initialize` with no instruction to rotate them. Settling every market
  depends on those exact council keys signing.
- **Why it matters:** If a council member's key is lost or compromised, there is no on-chain way to
  rotate it. With a 2-of-2 devnet council (or any N-of-N), losing one key makes the threshold
  unreachable and **permanently locks every in-flight position's escrow** — funds only release via
  `execute_settlement` at threshold. This is a fund-*availability* HIGH, not a theft vector. (Upside: a
  stolen `authority` key can do nothing today — zero admin blast radius.) Classified **launch** because
  the defect is governance/operational, not an exploit of program logic.
- **Recommended fix:** add `update_council(new_council, new_threshold)` gated by `has_one = authority` +
  `authority: Signer`, re-running `initialize`'s validity checks (size, strict-majority threshold ≤ size,
  no duplicates) on every update; hold `authority` under a Squads multisig.
- **Decision:** recommended-accept (human ratification pending).

### F-02 — Upgrade-authority disposition undocumented · HIGH · Launch Readiness
- **Where:** repo-level (`Anchor.toml`, deploy scripts) — no stated policy; the README defers mainnet
  "behind a security audit" but does not specify the upgrade-key plan.
- **What:** The program custodies user lamports. Branch 6.1 rates a live single-key upgrade authority on
  a value-holding *mainnet* program CRITICAL; here it is currently devnet and merely *unstated*, so it is
  rated HIGH (the risk is "defaults to a dev hot key unless deliberately set otherwise").
- **Why it matters:** Whoever holds the upgrade authority can replace the entire program with arbitrary
  code and drain every escrow — bypassing all the (good) on-chain checks. This single key dominates the
  risk profile regardless of code quality. Classified **launch** (branch 6).
- **Recommended fix:** before mainnet, set the upgrade authority to a Squads Protocol v4 M-of-N multisig,
  ideally behind a timelock, and publish the policy; or commit to `--final` (immutable) — but only after a
  pause/migration story exists, since bugs then become permanent.
- **Decision:** recommended-accept (pre-mainnet gate).

### F-03 — Single council member can grief parlay settlement · MEDIUM · Code Safety
- **Where:** `record_leg_result.rs:32-54` (any single council member, position still `Matched`, no
  settlement account in scope), vs `execute_settlement.rs:96-112` (legs re-checked against the voted
  outcome at execution).
- **What:** `sign_settlement` fixes the outcome at the first vote and accrues threshold votes. But
  `record_leg_result` is callable by *any one* council member at any time while the position is `Matched`
  — including *after* the threshold has voted `BettorWins`. Flipping any leg to `Lost` makes the
  `legs.iter().all(|l| l.result == Won)` check in `execute_settlement` fail (`ResultMismatch`), so the
  otherwise-valid settlement reverts and the escrow stays locked.
- **Why it matters:** A single rogue/compromised council member — *below* the honest-majority threshold
  the protocol is designed to tolerate — can unilaterally **block** a legitimate bettor-win parlay payout.
  That is a liveness/griefing attack the threshold model is supposed to prevent. It cannot *steal* funds
  (it can't force a payout to itself), but it can trap them. Classified **code** (on-chain logic
  reachable by a sub-threshold actor — branch 7/5), not launch: this is an exploit of the program, not a
  key-custody gap.
- **Recommended fix:** make leg results one-way (Pending→Won/Lost, reject re-recording — see F-05),
  and/or freeze `record_leg_result` once a `SettlementAccount` exists so leg facts are fixed before voting
  begins.
- **Decision:** recommended-accept.

### F-04 — `init` quote contradicts "improvable" design · MEDIUM · Code Safety
- **Where:** `submit_quote.rs:16-23` (`init`), vs `state.rs:128` comment ("one (improvable) quote per MM
  per RFQ").
- **What:** Because the quote PDA is created with `init` (not `init_if_needed`), a market maker who calls
  `submit_quote` a second time for the same RFQ hits an "account already in use" error. The documented
  intent — that an MM can *improve* their quote — is not achievable.
- **Why it matters:** Not a fund-loss bug, but a real functional/program defect: an MM is permanently
  locked to their first quoted odds for a given RFQ, with an opaque failure on retry. In a
  competitive-quoting product this undercuts the core mechanic. Classified **code** (program behavior,
  branch 4).
- **Recommended fix:** decide the intent. Either (a) keep `init`, drop the "improvable" claim, and surface
  a clean "already quoted" error in the client; or (b) switch to `init_if_needed` and overwrite the quote
  fields (guard `rfq_id`/`market_maker` so identity can't change), or add an explicit `update_quote`.
- **Decision:** recommended-accept.

### F-05 — Leg results lack finality · LOW · Code Safety
- **Where:** `record_leg_result.rs:54` overwrites `parlay_legs[idx].result` with no check that it was
  `Pending`.
- **What:** A leg can be recorded, then re-recorded to the opposite value repeatedly. Enables F-03 and
  general result churn.
- **Recommended fix:** require the current leg result is `Pending` before writing (one-way transition).
- **Decision:** recommended-defer (fold into the F-03 fix).

### F-06 — Settlement evidence is first-voter-only · LOW · Code Safety
- **Where:** `sign_settlement.rs:74-88` — only the `vote_count == 0` branch sets `evidence_uri`; later
  concurring voters cannot attach their own evidence.
- **What:** Weakens on-chain auditability of *why* the council chose an outcome; not a security issue.
- **Recommended fix:** optional — allow appending evidence, or accept the single-evidence design.
- **Decision:** recommended-defer.

### F-INFO-1 — Council is a fully trusted arbiter · INFO (trust assumption) · Launch
- **Where:** `sign_settlement.rs`, `record_leg_result.rs`, `execute_settlement.rs` — outcome and leg facts
  come entirely from council votes; no oracle/automated verification.
- **What:** The protocol resolves every market on council say-so. A colluding or compromised set of
  council members meeting the threshold can settle any position to any outcome and direct the payout to
  the (legitimately-named) winner of their chosen outcome.
- **Why it matters:** This is the *intended* trust model (README §"The council settles"), and the
  party-exclusion + no-double-vote + threshold checks are correctly implemented. Recorded as INFO, not a
  vuln, but it is the **dominant residual risk** and should be disclosed to users.
- **Mitigation:** multisig-grade key custody for council members; the roadmap's appeal layer / DAO
  governance; and (later) automated oracle resolution for `Standard` events to remove discretion.
- **Decision:** acknowledged (documented assumption).

### F-INFO-2 — Reputation metrics incomplete · INFO · Code Safety
- **Where:** `execute_settlement.rs:145-168` — `disputes_involved` is never incremented; bettor win/loss
  is not tracked symmetrically with MM win/loss.
- **What:** A metric-completeness gap in the reputation account, not a vulnerability.
- **Decision:** acknowledged (design note).

## Trust Assumptions (residual risk)

Even after all recommended fixes, users must trust:
1. **The council (threshold-of-N) is honest and available.** It is the sole arbiter of every outcome and
   parlay leg; a colluding threshold can mis-settle, and (pre-F-01-fix) a single lost key can lock funds.
   (F-INFO-1, F-01)
2. **The upgrade authority is not compromised.** Until set to a multisig/immutable, the program can be
   replaced and escrows drained. (F-02)
3. **Off-chain data feeds (api-football, The Odds API, Gemini) are advisory only.** The README confirms
   the Express proxy "never touches funds" and the program reads no off-chain data on the trusted path —
   the right posture (branch 9.4). No backend signer/relayer co-signs user-fund transactions (branch 9.5
   N/A); all fund-moving ixs are signed by the actual bettor / MM / cranker via the wallet.

## Invariants for Formal Verification (QEDGen / Lean-4 hand-off)

1. **Escrow conservation:** for any position, `payout_amount = stake + collateral`, and total lamports
   disbursed by `execute_settlement` ≤ the escrowed pot. Enforced at `execute_settlement.rs:90-93` + the
   per-outcome `move_lamports` calls (`execute_settlement.rs:118-132`).
2. **Non-negative vault:** `mm_vault` lamports never fall below its rent-exempt minimum after any
   `accept_quote`. Enforced via `util.rs:42-45` (`free_lamports`) + `accept_quote.rs:86-89`.
3. **Collateral > 0 (MM always at risk):** matched odds > 1.0x ⇒ collateral ≥ 1. Enforced at
   `submit_quote.rs:57-59`, `accept_quote.rs:81-82`.
4. **Single disbursement:** a settlement executes at most once (`executed` flag + position closed).
   Enforced at `execute_settlement.rs:74,143` + Anchor `close = bettor` on `position`.
5. **Deposit refunded exactly once:** `rfq.deposit` is zeroed after refund. Enforced at
   `execute_settlement.rs:135-139`.
